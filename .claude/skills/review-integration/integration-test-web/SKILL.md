# 集成文档实际测试 Skill

## When to Use
- 对集成文档进行**实际集成测试**，验证文档步骤是否真的可行
- 记录集成过程中遇到的实际卡点（命令失败、依赖错误、配置问题等）
- 生成 `test_issues` JSON，用于补充文档审查报告

## Instructions

**⚠️ 测试原则（最高优先级）**
- **严格按文档操作**：**必须严格按照文档中的步骤和示例代码来操作，不得自行修改、优化或添加功能**
- **真实环境测试**：在实际项目中按照文档步骤操作，不进行模拟
- **记录所有卡点**：无论问题大小，只要阻碍了集成流程，都要记录
- **提供解决方案**：对于发现的卡点，尽量提供可行的解决方案
- **区分文档问题与环境问题**：区分是文档本身的问题，还是测试环境的问题

**⚠️ 重要：代码使用规范**
- **必须使用文档中的示例代码**：不得修改文档中的代码示例（如添加错误处理、修改参数值、改变调用时机等）
- **严格按照文档步骤执行**：不得跳过步骤、不得改变步骤顺序、不得合并步骤
- **发现问题立即记录**：如果按照文档代码操作时遇到问题（如编译错误、运行时错误、逻辑错误等），必须记录为测试问题
- **禁止自行优化**：即使发现文档代码有潜在问题（如缺少错误处理、调用时机不对等），也要先按文档代码执行，然后将问题记录为文档问题

## 测试流程

### Step 1：解析集成文档

**目标**：提取文档中的关键信息，确定测试策略

**提取信息**：
1. **文档类型**：Web/Android/iOS/Flutter 等
2. **环境要求**：Node.js 版本、开发工具版本等
3. **集成步骤**：安装依赖、配置、初始化、API 调用等
4. **必要配置**：SDKAppID、SecretKey、UserSig 等

**⚠️ 重要**：测试所需的鉴权信息（SDKAppID、SecretKey 等）应从 `skills/_shared/authentication.md` 文件中获取，不要使用文档中示例的占位符值。

**输出**：测试计划（包含需要执行的步骤列表）

---

### Step 2：创建测试项目

**目标**：根据文档类型创建对应的测试项目

#### 2.1 Web 项目（React/Vue3）

**创建步骤**：
1. 创建项目目录：`integration-test-projects/web-{framework}-test`
2. 使用官方脚手架创建项目（如 `create-react-app`、`vite`）
3. 初始化 Git 仓库（可选）
4. 记录项目创建过程是否顺利

**检查项**：
- 项目创建是否成功
- 项目结构是否符合文档要求
- 是否需要额外的配置文件

**记录问题**：
- 项目创建失败 → 记录错误信息
- 脚手架版本不匹配 → 记录版本冲突
- 缺少必要文件 → 记录缺失的文件

#### 2.2 其他平台（后续扩展）
- Android：创建 Android Studio 项目
- iOS：创建 Xcode 项目
- Flutter：创建 Flutter 项目

---

### Step 3：环境准备验证

**目标**：验证文档中提到的环境要求是否满足

**检查项**：
1. **版本检查**
   - 检查 Node.js 版本是否符合要求
   - 检查其他工具版本（npm、yarn、pnpm 等）
   - 检查浏览器版本（Web 项目）

2. **依赖检查**
   - 检查系统依赖是否已安装
   - 检查全局工具是否可用

**记录问题**：
- 版本不匹配 → 记录当前版本 vs 要求版本
- 工具未安装 → 记录缺失的工具
- 版本检查命令失败 → 记录错误信息

---

### Step 4：执行集成步骤

**目标**：按照文档步骤逐一执行，记录每个步骤的执行结果

#### 4.1 安装依赖

**执行步骤**：
1. 根据文档说明，执行安装命令（npm install、pnpm install 等）
2. 检查安装是否成功
3. 检查依赖版本是否符合要求

**记录问题**：
- 安装命令失败 → 记录错误信息和失败原因
- 依赖版本冲突 → 记录冲突的包和版本
- 网络问题 → 记录网络错误（但不作为文档问题）
- 安装时间过长 → 记录但不作为问题（环境因素）

#### 4.2 配置 SDK

**执行步骤**：
1. 根据文档说明，找到配置文件
2. **从 `skills/_shared/authentication.md` 获取测试鉴权信息**：
   - 读取 `authentication.md` 文件
   - 获取对应的 SDKAppID（appid）和 SecretKey（key）
   - 根据文档说明生成 UserSig（如需要）
3. 填写必要的配置项（SDKAppID、UserSig 等）
4. 检查配置格式是否正确

**⚠️ 重要**：
- 必须使用 `authentication.md` 中的测试账号信息，不要使用文档中的占位符（如 `SDKAppID: 0`）
- 如果文档中未说明如何生成 UserSig，需要根据 SDK 的规范自行生成

**记录问题**：
- 配置文件不存在 → 记录文档中提到的路径 vs 实际路径
- 配置项格式错误 → 记录文档说明 vs 实际要求
- 配置项获取方式不明确 → 记录缺少的说明
- 配置后项目无法运行 → 记录错误信息
- authentication.md 中缺少必要信息 → 记录缺失的配置项

#### 4.3 初始化 SDK

**执行步骤**：
1. 根据文档说明，添加初始化代码
2. 检查导入语句是否正确
3. 检查初始化参数是否正确
4. 尝试运行项目，检查是否有编译错误

**记录问题**：
- 导入语句错误 → 记录文档中的导入方式 vs 实际可用的导入方式
- 初始化参数错误 → 记录参数类型/格式问题
- 编译错误 → 记录具体的错误信息
- 运行时错误 → 记录错误堆栈

#### 4.4 API 调用测试

**执行步骤**：
1. **严格按照文档中的代码示例**，添加 API 调用代码（不得修改、优化或添加功能）
2. 检查 API 是否存在
3. 检查参数是否正确
4. 检查回调处理是否正确
5. 检查代码逻辑是否正确（如调用时机、执行顺序等）
6. 尝试运行，检查是否有错误

**⚠️ 重要**：
- **必须使用文档中的示例代码**，不得自行修改
- **严格按照文档中的调用方式**，不得改变调用时机或顺序
- **如果文档代码有问题**（如调用时机不对、缺少错误处理等），也要先按文档执行，然后记录为文档问题

**记录问题**：
- API 不存在 → 记录文档中的 API 名称 vs 实际可用的 API
- 参数类型错误 → 记录文档说明 vs 实际要求
- 回调处理错误 → 记录回调参数问题
- 运行时错误 → 记录错误信息
- **代码逻辑错误** → 记录文档代码中的逻辑问题（如调用时机不对、执行顺序错误等）

---

### Step 5：验证集成结果

**目标**：验证集成是否成功，功能是否可用

**检查项**：
1. **编译检查**
   - 项目是否能成功编译
   - 是否有编译警告（记录但不作为问题）

2. **运行检查**
   - 项目是否能成功运行
   - 是否有运行时错误

3. **功能检查**
   - 核心功能是否能正常使用
   - 是否能达到文档中描述的效果

**记录问题**：
- 编译失败 → 记录编译错误
- 运行失败 → 记录运行时错误
- 功能不可用 → 记录功能测试结果
- 效果不符合预期 → 记录实际效果 vs 文档描述

---

### Step 6：生成测试报告

**目标**：汇总所有测试过程中发现的问题，生成 `test_issues` JSON

**报告内容**：
1. **测试环境信息**
   - 测试时间
   - 测试环境（操作系统、工具版本等）
   - 测试项目路径

2. **测试结果**
   - 集成是否成功
   - 功能是否可用
   - 总体评价

3. **发现的问题**
   - 每个问题的详细信息
   - 问题分类（文档问题 vs 环境问题）
   - 问题严重程度
   - 解决方案建议

---

## Output Format（强制）

请输出 JSON 格式，如果没有问题则输出空数组 `[]`。

```json
{
  "test_issues": [
    {
      "step": "安装依赖/配置 SDK/初始化 SDK/API 调用/验证集成",
      "type": "命令失败/依赖错误/配置错误/代码错误/功能不可用/文档不准确",
      "severity": "P0/P1/P2",
      "location": "文档中的具体位置（章节、代码行等）",
      "description": "问题描述（实际遇到的问题）",
      "error_message": "具体的错误信息（如有）",
      "expected": "文档中描述的行为",
      "actual": "实际发生的行为",
      "solution": "解决方案建议",
      "is_document_issue": true/false,
      "test_environment": {
        "os": "操作系统",
        "node_version": "Node.js 版本",
        "npm_version": "npm 版本",
        "project_path": "测试项目路径"
      }
    }
  ],
  "test_summary": {
    "test_time": "测试时间",
    "integration_success": true/false,
    "function_usable": true/false,
    "total_issues": 0,
    "p0_count": 0,
    "p1_count": 0,
    "p2_count": 0
  }
}
```

**Severity 判定标准**：
- **P0**：导致**无法完成集成**或**集成后无法使用**（如命令失败、代码错误、API 不存在）
- **P1**：导致**集成困难**或**需要额外摸索**（如配置不明确、版本不匹配、文档不准确）
- **P2**：不影响集成，但**影响体验**（如缺少提示信息、步骤顺序不合理）

**is_document_issue 判定标准**：
- **true**：文档本身的问题（如步骤错误、代码示例错误、配置说明不准确）
- **false**：环境问题或非文档问题（如网络问题、系统配置问题）

## 测试项目模板

### Web React 测试项目

**项目结构**：
```
integration-test-projects/
└── web-react-test/
    ├── package.json
    ├── vite.config.js (或 webpack.config.js)
    ├── src/
    │   ├── App.jsx
    │   ├── main.jsx
    │   └── config.js (配置文件)
    └── README.md (测试记录)
```

**创建命令**：
```bash
# 使用 Vite 创建 React 项目
npm create vite@latest web-react-test -- --template react

# 或使用 Create React App
npx create-react-app web-react-test
```

### Web Vue3 测试项目

**项目结构**：
```
integration-test-projects/
└── web-vue3-test/
    ├── package.json
    ├── vite.config.js
    ├── src/
    │   ├── App.vue
    │   ├── main.js
    │   └── config.js (配置文件)
    └── README.md (测试记录)
```

**创建命令**：
```bash
# 使用 Vite 创建 Vue3 项目
npm create vite@latest web-vue3-test -- --template vue
```

## 注意事项

1. **测试环境隔离**：每个文档的测试应在独立的项目中进行，避免相互影响
2. **清理测试项目**：测试完成后，可以选择保留或删除测试项目（建议保留用于复现问题）
3. **鉴权信息获取**：**必须**从 `skills/_shared/authentication.md` 文件中获取测试所需的鉴权信息（SDKAppID、SecretKey 等），不要使用文档中的占位符值
4. **敏感信息处理**：测试中使用的 SDKAppID、SecretKey 等应使用测试账号，不要使用生产环境信息
5. **版本记录**：详细记录测试时使用的所有工具版本，便于复现问题
6. **错误截图**：对于重要的错误，建议保存截图或日志文件
