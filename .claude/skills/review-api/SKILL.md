# API 参考文档审查 Skill

## When to Use

- 审查 RESTful API 参考文档
- 审查包含 API 签名、参数表、请求/响应示例的文档
- 文档目标是让开发者能够**仅凭文档**正确实现并使用 API，无需询问接口作者

## Instructions

**⚠️ 零误报原则（最高优先级）**
- **误报会让用户失去信任**，这比漏报更严重！
- 每个问题必须有确凿证据，不能基于推测、习惯、直觉判断
- 不确定的问题必须标记为 `【需人工确认】`
- **核心原则**：如果一个问题在不联系文档作者的情况下，开发者仍然可以 100% 正确实现接口，请不要输出

---

### Step 0：执行共享层审查（强制）

在开始任何审核前，**必须完整执行以下共享审查规则**：

4. `language-review.md`
5. `terms-review.md`

> 若共享层存在 FAIL 项，需记录，但仍继续执行后续步骤。

---

### Step 1：检查示例与描述一致性

检查请求/响应示例中的字段、参数、结构是否与参数表或文字说明一致。

**必须标记**：
- 文档描述了字段，但示例中不存在 → **P0**
- 示例中出现字段，但文档未说明其含义或类型 → **P0**
- 示例中的字段值与文档描述的类型/格式不匹配 → **P0**
- 示例中的嵌套结构与文档描述不匹配（如文档说 Object，示例是 Array）→ **P0**

**不标记**：
- 示例是简化版本，文档已说明"简化示例"或"仅展示关键字段"
- 可选字段未出现，文档已说明为可选

---

### Step 2：检查必填/可选规则

检查参数标注的必填/可选规则是否与实际使用一致。

**必须标记**：
- 参数标注为必填，但示例中缺失 → **P0**
- 参数标注为必填，但文档说明该参数有默认值（逻辑矛盾）→ **P0**
- 参数标注为可选，但缺少该参数会导致请求失败（逻辑矛盾）→ **P0**

**不标记**：
- 可选参数在示例中未出现，文档已说明为可选
- 参数有条件必填，文档已说明条件

---

### Step 3：检查类型和结构一致性

检查同一字段在不同位置的类型、结构描述是否一致。

**必须标记**：
- 参数表中标注为 String，示例中是 Number → **P0**
- 参数表中标注为 Object/Array，示例中是其他类型 → **P0**
- 嵌套结构在示例与说明中不匹配（如文档说 `user.name`，示例是 `userName`）→ **P0**

**不标记**：
- 文档说明了类型转换规则或不同场景下的不同结构

---

### Step 4：检查枚举值和取值范围

检查枚举值或取值范围在不同位置是否一致。

**必须标记**：
- 枚举值在不同位置不一致（如参数表说 `status: active|inactive`，示例中是 `status: enabled`）→ **P0**
- 示例中出现文档未定义的枚举值 → **P0**
- 文档说明的取值范围与示例中的值不匹配 → **P0**

**不标记**：
- 文档说明了枚举值的扩展性或不同版本支持不同的枚举值

---

### Step 5：检查命名一致性

检查同一含义的字段在不同位置是否使用相同的字段名。

**必须标记**：
- 参数表中字段名是 `user_id`，示例中是 `userId`（snake_case / camelCase 混用，且文档未说明转换规则）→ **P0**
- 请求参数和响应中使用不同字段名表示同一含义，且文档未说明 → **P0**

**不标记**：
- 文档明确说明了命名转换规则（如"请求使用 camelCase，响应使用 snake_case"）

---

### Step 6：检查关键信息完整性

检查开发者是否能够仅凭文档正确实现接口。

**必须标记**：
- 无法确定如何正确构造请求（缺少请求方法、URL、请求头、请求体格式说明）→ **P0**
- 缺少认证方式说明（如需要 Token 但未说明如何获取）→ **P0**
- 无法正确解析响应（缺少响应格式说明、字段含义说明）→ **P0**
- 缺少成功/失败判断依据（如状态码、错误码说明）→ **P0**
- 缺少错误码说明或错误处理示例 → **P1**

**不标记**：
- 文档引用了其他文档说明认证方式或错误处理信息

---

### Step 7：检查描述歧义

检查文档描述是否存在歧义，且会导致错误实现。

**必须标记**：
- 字段含义存在多种合理但结果不同的理解（如"时间"未说明是时间戳还是字符串）→ **P0**
- 字段用途描述模糊，无法确定如何使用 → **P0**
- 不同段落对同一行为描述不一致 → **P0**

**不标记**：
- 描述有多种理解方式，但都能正确实现接口
- 描述不够详细，但不影响正确实现

---

## 不检查的内容

以下内容一律不输出：
- ❌ 命名或风格优化建议
- ❌ 文案表达是否优雅
- ❌ 性能、安全、架构层面的建议
- ❌ "可以更清楚"但不影响正确实现的问题
- ❌ 主观偏好或最佳实践类建议

---

## Output Format（强制）

请输出 JSON 格式，如果没有问题则输出空数组 `[]`。

**⚠️ 重要说明**：此 JSON 输出用于在内存中汇总审查结果，**不单独保存为文件**。最终会生成汇总的 JSON 文件（包含所有审查模块结果）和 Markdown 报告。

```json
{
  "api_issues": [
    {
      "type": "示例与描述不一致/字段缺失或多余/必填可选规则冲突/类型结构矛盾/枚举值不一致/命名不一致/关键信息缺失/描述存在歧义",
      "severity": "P0/P1",
      "category": "api",
      "location": "问题所在位置（如"参数表"、"请求示例"、"响应示例"）",
      "original": "有问题的原文或示例",
      "description": "问题描述（说明为什么这是问题，会导致什么错误）",
      "suggestion": "修改建议（具体的修改方案）"
    }
  ]
}
```

---

## Severity 判断标准

- **P0（严重）**：必然导致实现错误、无法实现或必须反复试错的问题
- **P1（警告）**：影响使用体验但不影响基本功能的问题（如错误处理信息不完整）

---

## 注意事项

1. **聚焦可操作性**：每个问题都必须能给出具体的修改建议。
2. **避免重复**：共享层已检查的问题不要重复输出。
3. **严格以文档为准**：不得基于假设补全文档缺失信息，不得为文档作者"脑补合理实现"。
4. **避免过度审查**：只输出"必然导致实现错误、无法实现或必须反复试错"的问题。
5. **不确定的问题**：标记为 `【需人工确认】`，在 description 中说明不确定的原因。
