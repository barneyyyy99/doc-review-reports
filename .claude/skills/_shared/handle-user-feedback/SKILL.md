# 用户反馈处理 Skill

## When to Use

- 用户在对话中指出审查报告的问题（误报、漏报、描述不准确、优先级不当）
- 用户明确要求修复报告或更新规则

## Instructions

**⚠️ 核心原则**
- **逐个确认**：每个修改操作都需要用户明确确认
- **三件事必做**：修改报告、更新规则、记录 CHANGELOG
- **统计一致性**：修改后必须重新统计，确保数据准确

---

## 工作流程

### Step 1：识别反馈类型

识别用户反馈的问题类型：
- **误报**："不是问题"、"误报"、"不应该标记"
- **漏报**："漏报了"、"应该检查"、"没有发现"
- **描述不准确**："描述不对"、"建议不够具体"
- **优先级不当**："不是P0"、"应该是P1"

### Step 2：定位报告中的问题

通过以下方式定位：
- 问题编号（"第X个问题"、"第X个P0问题"）
- 问题描述关键词
- 位置信息（"XX章节"）
- 问题类型标签（"[格式]"、"[代码]"）

找到问题后，展示完整内容确认。

### Step 3：生成修改计划并逐个确认

为每个问题生成修改计划，逐个展示并等待用户确认：

**修改操作类型**：
1. **删除问题（误报）**：移除整个问题块，更新统计
2. **添加问题（漏报）**：在对应优先级章节添加，使用标准格式
3. **修改问题（描述不准确）**：修改描述、建议或位置
4. **调整优先级**：修改优先级并移动问题位置，重新编号

**确认流程**：
1. 展示修改计划（操作类型、涉及内容、原因）
2. 等待用户确认（"确认"/"同意" → 执行，"拒绝"/"跳过" → 跳过）
3. 执行修改（使用 `search_replace` 精确修改）
4. 更新统计表格（重新统计，确保 P0 + P1 + P2 = 总计）

### Step 4：更新统计表格

修改完成后必须：
1. 重新统计每个优先级的问题数量
2. 更新报告头部的统计表格
3. 验证一致性（统计表数量 = 实际问题数量）

### Step 5：更新审查规则

分析问题根源，更新对应规则（复用 `update-review-rules` 逻辑）：

1. **分析问题根源**：为什么误报/漏报/描述不准确？
2. **确定影响范围**：共享层/业务层/报告生成规则
3. **生成规则变更 diff**：展示清晰的 diff，说明原因和影响
4. **等待用户确认**：询问是否需要更新规则
5. **应用规则更新**：用户确认后修改规则文件

### Step 6：记录到 CHANGELOG

在 CHANGELOG.md 开头添加变更记录：

```markdown
## YYYY-MM-DD

### [报告修改] 文档名称
**原因**: 用户反馈：XXX  
**修改内容**: 
- 删除误报：问题X（原因为XXX）
- 添加漏报：问题Y（位置：XXX，优先级：P1）
- 修正描述：问题Z（原描述→新描述）
- 调整优先级：问题W（P0→P1）

### [规则更新] 影响层级 - 文件路径
**原因**: 用户反馈：XXX  
**影响**: XXX  
**变更**: XXX
```

**记录要点**：
- 必须记录所有报告修改和规则更新
- 使用 YYYY-MM-DD 格式，最新记录在最上面
- 如果当天已有记录，合并到同一天

---

## 注意事项

1. **保持格式一致**：修改报告时保持 Markdown 格式、问题编号连续
2. **统计一致性**：每次修改后必须重新统计，确保数据准确
3. **问题编号**：删除/添加/移动问题后需要重新编号
4. **规则更新可选**：如果用户只要求修改报告，可以询问是否需要更新规则
5. **CHANGELOG 完整性**：必须记录所有修改操作

---

## Output Format

对话式输出，包括：
1. 修改计划（展示要修改的内容）
2. 确认请求（等待用户回复）
3. 修改完成反馈（说明已修改的内容）
4. 规则更新建议（如果需要）
5. CHANGELOG 记录说明
