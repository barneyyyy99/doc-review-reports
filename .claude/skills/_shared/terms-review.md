# 审查目标
检查文档中与文档介绍的功能强相关的术语在文档中使用的一致性，确保同一概念在整个文档中使用统一的术语表达。

# 审查原则

**核心原则**：仅检查与文档核心功能**强相关**的术语一致性，避免过度审查。仅根据#审查规则来检查文档中的术语使用。

> **优先级标准**：本文档中的 P0/P1/P2 标注遵循 `skills/_shared/severity-standards.md` 定义的统一标准。

**⚠️ 零误报原则**：
- 误报会让用户失去信任，这比漏报更严重！
- 每个问题必须有确凿证据，不能基于推测、习惯、直觉判断
- 不确定的问题必须标记为 `【需人工确认】`，不能作为确定性问题输出

# 审查规则

## 1. 核心产品/组件名称一致性

检查文档中核心产品名、组件名是否一致使用。

**检查项**：
- 产品名称（如 `TUILiveKit`、`LiveGift`）在全文中是否统一
- 组件名称在正文描述和代码示例中是否一致
- 是否有混用（如"直播送礼组件"和"礼物组件"混用）

**必须标记的情况**：
- ❌ 文档开头说"接入 **LiveGift** 组件"，后文说"使用 **礼物组件**"（未明确是同一组件）
- ❌ 代码示例中使用 `LiveGift`，但正文描述说"使用 `GiftPanel` 组件"
- ❌ 产品名 `TUILiveKit` 在文档中有时写作 `TUI LiveKit`（空格不一致）

**不标记的情况**：
- ✅ 首次介绍时使用全称，后续使用简称（如"TUILiveKit 直播送礼组件" → "直播送礼组件"）
- ✅ 在代码注释中使用不同的命名（代码有自己的命名规范）
- ✅ 引用其他文档时使用了不同的术语（引用内容不受本文档控制）
---

## 2. 功能概念术语一致性

检查文档中描述的核心功能概念是否使用统一术语。

**检查项**：
- 功能名称是否一致（如"礼物系统"、"礼物功能"、"礼物服务"）
- 功能模块名称是否一致（如"礼物列表"、"礼物发送"、"礼物播放"）
- 关键操作术语是否一致（如"发送礼物"、"赠送礼物"、"打赏礼物"）

**必须标记的情况**：
- ❌ 文档标题说"礼物系统"，正文中混用"礼物功能"、"礼物服务"（未说明是同一概念）
- ❌ 功能描述中说"礼物列表"，代码注释中说"礼品列表"
- ❌ 操作描述中说"发送礼物"，后文说"赠送礼物"（未说明是同一操作）

**不标记的情况**：
- ✅ 在不同上下文中使用不同的表达（如"礼物系统"指整体功能，"礼物功能"指具体能力）
- ✅ 引用外部文档或 API 时使用了不同的术语
- ✅ 口语化表达（如"送礼物"）和正式术语（如"发送礼物"）混用，但含义明确

**示例**：
```
❌ 错误案例：
- 第1行："接入礼物系统"
- 第43行："使用礼物功能需要开通服务"
- 第101行："礼物服务后端对接"
→ 问题："礼物系统"、"礼物功能"、"礼物服务"混用，未明确是否为同一概念

✅ 正确案例：
- 全文统一使用"礼物系统"
→ 或：明确说明"礼物系统（以下简称礼物功能）"
```

---

## 3. 技术术语一致性

检查文档中技术术语的使用是否一致。

**检查项**：
- 技术标准名称是否一致（如 `SVG`、`MP4`、`SVGA`、`PAG`）
- 技术概念是否一致（如"全屏播放"、"全屏展示"）
- 平台/框架名称是否一致（如 `Vue3`、`Vue 3`、`Vue.js 3`）

**必须标记的情况**：
- ❌ 文档中说"支持 SVG 格式"，后文说"支持 svg 动画"（大小写不一致）
- ❌ 文档中说"Vue3 项目"，后文说"Vue.js 3 项目"（未说明是同一框架）
- ❌ 文档中说"全屏播放"，后文说"全屏展示"（未明确是否为同一概念）

**不标记的情况**：
- ✅ 技术术语在不同上下文中使用不同的表达（如"SVG 动画"和"SVG 格式文件"）
- ✅ 引用外部文档或标准时使用了不同的术语
- ✅ 代码中的技术术语和正文描述不同（代码有自己的命名规范）

**示例**：
```
❌ 错误案例：
- 第29行："支持 SVG 2D、SVG 3D 和 MP4 视频礼物"
- 第110行："如果 svga 无法满足需求"
→ 问题：SVG 和 svga 大小写不一致（SVGA 是特定技术，不是 SVG 的小写）

✅ 正确案例：
- 全文统一使用 "SVG" 和 "SVGA"
```

---

## 5. 检查方法

**执行步骤**：

1. **提取核心术语**：
   - 从文档标题、开头介绍中提取核心产品名、组件名
   - 从功能描述中提取功能概念术语
   - 从代码示例中提取 API/类名

2. **全文搜索验证**：
   - 对每个核心术语，搜索全文中的所有出现位置
   - 检查是否有不一致的表达
   - 确认不一致是否为误报（如上下文不同、引用外部内容等）

3. **判断是否为问题**：
   - 如果同一概念使用了不同术语，且可能造成混淆 → 标记为问题
   - 如果术语不同但含义明确（如全称和简称） → 不标记
   - 如果不确定 → 标记为【需人工确认】

**注意事项**：
- 只检查与文档核心功能**强相关**的术语
- 不要检查通用词汇（如"用户"、"数据"等）
- 不要检查代码中的变量名、函数名（代码有自己的命名规范）
- 重点关注可能造成开发者混淆的术语不一致

# 输出格式

请输出 JSON 格式，如果没有问题则输出空数组 `[]`。

**⚠️ 重要说明**：此 JSON 输出用于在内存中汇总审查结果，**不单独保存为文件**。最终会生成汇总的 JSON 文件（包含所有审查模块结果）和 Markdown 报告。

```json
{
  "terms_issues": [
    {
      "type": "产品/组件名称不一致/功能概念术语不一致/技术术语不一致",
      "severity": "P2",
      "category": "terms",
      "location": "问题所在位置（如"第X行"或"XX章节"）",
      "original": "不一致的术语表达（列出所有不一致的表达）",
      "description": "问题描述（说明为什么这是问题，可能造成什么混淆）",
      "suggestion": "修改建议（统一使用哪个术语）"
    }
  ]
}
```

**注意事项**：
1. **severity 判断标准**：
   - **P2（警告）**：术语不一致可能造成开发者混淆，影响理解
   - 术语一致性问题通常不标记为 P0（不会导致代码无法运行）
2. 对于同一类型的问题，如果文档中存在超过 3 处，则仅输出 3 处错误，"suggestion"中增加"全文中有多处，请批量修改"。
3. **聚焦可操作性**：每个问题都必须能给出具体的修改建议，说明应该统一使用哪个术语。
4. **避免过度审查**：
   - 不要检查通用词汇的一致性
   - 不要检查代码中的变量名、函数名
   - 只关注可能造成混淆的核心术语不一致
