# 审查目标
按照下列下列审查规则来检查技术文档中是否有不符合规则的表达，除非另有说明不需要报告，否则不符合审查规则的问题都需要报告

# 审查原则

**核心原则**：按照下列审查规则来审核技术文档中的语言问题，确保文档中的表述清晰、直接，符合开发者认知。

> **优先级标准**：本文档中的 P0/P1/P2 标注遵循 `skills/_shared/severity-standards.md` 定义的统一标准。

**⚠️ 零误报原则**：
- 误报会让用户失去信任，特别是错别字的误报会严重影响专业性
- 标记错别字前必须确认无误，不能把生僻词、专业术语、品牌名误判为错别字
- 不确定的问题不要输出

# 审查规则：

1. 禁止口语化表达或者行业黑话
文档中禁止出现口语化表达，比如：
- 挺好用
- 介绍一下
- 非常有用
- 踢人（应该用“移出房间”）

2. 语言简洁清晰：技术文档应使用简洁、直接、客观的表达方式，避免不必要的冗余词汇。

3. 避免过度使用"您"、"您可以"、"您只需"等表达会使文档显得啰嗦，降低阅读效率。

注意：关于“您”的过渡滥用，可统计"您"的使用频率来判断：
每 100 字中“您”出现次数
  - < 3 次：正常
  - 3-5 次：建议优化
  - > 5 次：需要优化

**执行要点：**
- 优先关注高频出现的位置（文档开头、步骤说明、段落开头）
- 重点检查是否有"您"连续出现的情况
- 规则 3 中审核出来的问题都算是 P2（风格优化，非必须修改的错误）

4. 章节标题规范：
- 对于 demo、集成、功能文档中的**实现**章节中的部分，章节标题应该使用"动词+宾语"的结构
- 标题应该直观、易懂，避免出现生僻的技术词汇
```示例
### 步骤 3：配置 live-list 文件 应改为"步骤 3：跳转至直播列表页面"
```
- 标题中禁止出现"您的"

- 同一个章节标题下的子标题，结构需尽量保持一致
```错误案例：
## 实现通话
### 1. 初始化 SDK
### 2. 音频配置（应该是"配置音频"，跟前面的标题保持一致，使用动词+宾语的结构）
```

```错误案例：
### 步骤6：参与者信息更新（错误：宾语+动词，应改为"更新参与者信息"）
### 步骤7：邀请打开设备（正确：动词+宾语）
```

- 标题禁止使用完整的句子（FAQ 除外）

**检查方法**：
- 对于"实现步骤"等章节，必须将所有同级标题列出来横向对比
- 逐个检查是否符合"动词+宾语"结构
- 检查结构是否一致（如出现"宾语+动词"倒置结构）


5. 视角一致性：技术文档的目标读者是**开发者**，不是最终用户。要保持开发者视角，避免视角混乱。

**错误案例：**
```
❌ "控制本地音频上行的采集音量大小，即观众听到的您的声音大小。"
问题：这里的"您"指的是最终用户（主播），但文档是写给开发者的，造成视角混乱。

✅ 正确："控制本地音频上行的采集音量大小，即观众听到的主播声音大小。"
或："控制本地音频上行的采集音量大小，即观众听到的用户声音大小。"
```

**检查要点：**
- "您"只能用于指代开发者本人的开发行为（如"您可以调用该方法"）
- 混合视角是 P1 问题（影响理解），需要修复

6. 禁止出现错别字
注意：错别字属于 **P0** 问题（影响专业性，降低用户信任）。

# 输出格式
请输出 JSON 格式，如果没有问题则输出空数组 `[]`。

**⚠️ 重要说明**：此 JSON 输出用于在内存中汇总审查结果，**不单独保存为文件**。最终会生成汇总的 JSON 文件（包含所有审查模块结果）和 Markdown 报告。

{
  "language_issues": [
    {
      "type": "口语化/简洁性/标题规范/视角混乱/错别字/API链接错误",
      "severity": "P1/P2",
      "category": "language",
      "location": "所在行或章节",
      "original": "原文内容",
      "description": "问题描述",
      "suggestion": "修改后的建议"
    }
  ]
}

注意事项：
1. **severity 判断标准**：
   - **P0（严重）**：错别字（影响专业性，降低用户信任）
   - **P1（警告）**：视角混乱问题（影响理解，必须修改）
   - **P2（建议）**：其他语言规范问题（风格优化，如过度使用"您"、简洁性等）
2. 对于同一类型的问题，如果文档中存在超过3处及以上问题，则仅输出 3 处错误，"suggestion"中增加"全文中有多处，请批量修改"。
3. **聚焦可操作性**：每个问题都必须能给出具体的修改建议，而不是泛泛而谈。
