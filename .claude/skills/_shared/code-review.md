# 审查目标
审查技术文档中的代码片段（Code Blocks）的正确性，确保代码示例准确、可用，不会误导外部开发者。

# 审查原则

**核心原则**：只关注**错误**和**会导致用户无法运行或理解代码的问题**。不要过度干涉代码优化和最佳实践，除非存在巨大隐患。仅根据#审查规则来检查文档中的示例代码。

> **优先级标准**：本文档中的 P0/P1/P2 标注遵循 `skills/_shared/severity-standards.md` 定义的统一标准。

# 审查方式

## Step 1：检查文档中的示例代码是否存在下列问题，如果文档中存在下列问题，除非有特殊说明，否则都要报告


1. **语法错误** → **P0**
   - 检查代码是否存在明显的语法错误（如缺少分号、括号不匹配、中英文符号混用、缩进错误导致无法运行、API 使用方式错误导致编译失败）
   - **理由**：导致代码无法编译或运行

2. **API 拼写错误** → **P0**
   - 检查全文中 API 名称、参数、类名等**名称拼写**是否正确、一致（区分大小写）
   - 示例：`UIColro`（应为 `UIColor`）
   - 示例：`getData()`（文档其他地方是 `fetchData()`）
   - **注意**：这里仅指名称拼写，不包括 API 使用方式错误（使用方式错误归类为"语法错误"）
   - **理由**：导致代码无法编译

3. **参数类型不匹配** → **P0**
   - 检查示例代码中传入参数类型是否与上下文描述或 API 定义明显不匹配（如文档说传 String 但代码传了 Int）
   - **理由**：导致运行时错误或理解错误

4. **正文与代码矛盾** → **P0**（核心功能）/ **P1**（注释错误）
   - 检查代码与正文描述的逻辑是否存在明显矛盾（如正文说调用方法 A，但代码调用的是方法 B）
   - **P0 场景**：正文描述与代码逻辑完全相反，会导致功能无法实现
   - **P1 场景**：注释与参数名矛盾（如 `autoEnableMicrophone` 注释说"摄像头"），参数名本身清晰
   - **理由**：造成理解混乱，P0 会导致实现错误，P1 仅影响理解

5. **导入/依赖缺失** → **P0**
   - 使用的类/函数/库未导入（前面已导入过的除外）
   - **理由**：代码无法编译或运行

6. **代码片段不完整** → **P0**（关键缺失）/ **不标记**（正常简化）
   - ❌ **必须标记为 P0**：
     - 代码声称"完整示例/可运行代码"，但缺少必需的初始化/配置
     - 关键变量/对象完全未提及来源（无注释、无前文说明、无上下文）
     - 调用了自定义方法/函数，但该方法完全未定义且未说明
   - ✅ **不要标记**：
     - 占位符+注释说明来源（如 `userId // 从登录回调获取`）
     - 前文已展示的对象/变量，后续直接使用
     - 聚焦核心 API，省略业务逻辑细节（错误处理、UI 代码等）
   - ⚠️ **检查前必须**：搜索全文确认是否在其他位置定义，不确定则标记【需人工确认】
   - **理由**：P0 场景会导致代码无法运行

7. **硬编码敏感信息** → **P0**
   - 真实密钥、密码、Token、内部 IP 等
   - **理由**：存在安全风险

8. **代码块语言标记错误** → **P1**
   - 代码块内容与声明的语言不匹配（如 Kotlin 代码标记为 `java`）
   - 如果存在多个此类问题只许报一次一场并标出多个位置
   - **理由**：影响语法高亮和阅读体验，但不影响代码功能

9. **错误处理缺失** → **P0**（正文要求但缺失）/ **不标记**（示例简化）
   - ❌ **必须标记为 P0**：
     - 文档正文明确要求处理错误（如"必须捕获异常"），但代码完全没有实现
     - 关键操作（网络请求、文件操作、数据库操作）无任何错误处理，且会导致程序崩溃
   - ✅ **不要标记**：
     - 文档只是展示核心 API 用法，省略错误处理（这是正常的简化示例）
     - 代码已有基本的错误处理（即使不完善）
   - **理由**：P0 场景会导致程序崩溃或与描述矛盾

10. **资源泄漏** → **P0**（明显泄漏）/ **不标记**（自动管理）
    - ❌ **必须标记为 P0**：
      - 打开文件/连接/流后未关闭，且文档未说明这是简化示例
      - 明显的内存泄漏（如循环中不断创建对象但不释放）
    - ✅ **不要标记**：
      - 使用了自动资源管理（如 Java 的 try-with-resources、Python 的 with 语句）
      - 文档明确说明"实际使用需要关闭资源"
      - 资源由框架/SDK 自动管理
    - **理由**：会导致资源耗尽


## Step 2：输出前检查

1. **重新审视你输出的问题**，再次确认是否真的有错误：
   - 如果的确是错误，继续输出
   - 如果是误报，删除该条
   - 如果不确定，在 type 或 description 字段中标记【需人工确认】
   
   **需人工确认的典型场景**：
   - 项目可能有自定义扩展（如自定义的 `UIColor(hex:)` 初始化器）
   - 代码可能使用了文档未提及的三方库
   - 语法在某些语言版本中可能合法但不确定项目使用哪个版本

2. **代码结构检查**：在标记"代码片段不完整"前，必须：
   - 仔细检查代码的完整结构（类的所有方法、字段等）
   - 确认方法是否定义在类的其他位置
   - 如果代码结构是正确的，即使看起来"不完整"，也不要标记为问题

3. **避免过度要求**：
   - 不要要求文档重复说明代码示例中已经展示的内容
   - 不要要求文档说明所有可能的边界情况，除非这些情况会导致代码无法运行
   - 不要要求文档展示所有实现细节（如占位符变量的完整取值逻辑），这是开发者的正常编程任务
   
4. **跨代码块引用判断**：
   - 如果前面的代码块展示了回调参数对象（如 `onEvent: (user)`），后续代码块使用该对象的字段（如 `user.id`）时，即使使用占位符+注释的方式，也不应标记为问题
   - 开发者能够根据上下文和注释推断出变量的取值方式，这是正常的代码阅读能力
   - **标记问题的唯一理由**：上下文中完全没有出现过该对象，且注释也没有说明来源

## 不检查的内容（避免过度干涉）
- 代码风格和编码规范（命名规范、代码格式等）
- 性能优化建议（除非存在明显会导致系统崩溃的性能问题）
- 边界条件处理（除非文档明确要求处理但代码完全没有）
- 异步/并发的最佳实践（除非代码明显错误导致无法运行）
- 代码重构建议
- 设计模式建议
