# 审查目标：检查文档的排版格式是否满足下列规范，仅报告有问题的部分：

# 核心原则：仅根据#审查规范来检查文档中的格式。

> **优先级标准**：本文档中的 P0/P1/P2 标注遵循 `skills/_shared/severity-standards.md` 定义的统一标准。

# 审查规范
1. 空格检查：正文（非代码块或代码注释）中，中文字符与英文字符（英文字母、英文单词）之间需要有一个空格。**数字与中文之间不需要加空格**（如 "5.0及以上"、"步骤1" 不视为错误）。**加粗英文后直接跟中文不需要加空格**（如 `**LiveKit**提供` 不视为错误）。
```正确案例
1. Chat 服务提供下列功能。
2. 确保已经实现 [login](#link) 步骤。
3. 调用 `join` 加入房间。
4. Android 5.0及以上版本。（数字与中文之间可不加空格）
5. **LiveKit**提供弹幕功能。（加粗英文后直接跟中文，不需要加空格）
```

```错误案例
1. LiveKit提供弹幕功能。（正确应为“LiveKit 提供弹幕功能”）
2. 通过`call`方法实现通话。（正确应为“通过 `call` 方法实现通话”）
```

2. 代码格式：正文描述中的 API 名、类名、参数名、文件路径需要使用了 Markdown 行内代码格式（即被 `` 包裹）。
不强制所有技术/代码内容都必须用反引号，而检查同一技术术语在全文的行内代码格式是否一致，不得出现前文使用了代码格式而后文没有使用
-触发条件
1 对某个 token，只有同时满足以下条件才会报：
2 全文某处已经把它用反引号包裹过：文档里存在 `token`
3 当前行出现了同一个 token，但没有被反引号包裹
4 当前行处于“技术语境”（用于降噪）：当前行包含至少一个提示词，例如：“参数/字段/配置/文件/路径/变量/值/打开/修改/填入/获取/调用/返回/类型”
本身是明显路径/文件名/以 ID 结尾等强技术形态
满足以上条件 → 输出一条 type="代码格式" 的问题，建议把当前行的 token 用反引号包裹。
-明确不检查/不强制的情况
不强制所有 API/类名/参数名都包反引号（只做“一致性”）
函数/指令形态不强制：如果 token 紧跟 (（例如 login(）会跳过（视作函数调用）
命令行/代码行（如 npm install、import ...、a = b）会跳过
HTML/表格格式内容（如 <table>, colSpan）会被忽略，不参与判断
如果某 token 全文从未以反引号出现过 → 不会要求它在其他地方补反引号
只允许报至多一次错误并标出具体位置，不许反复报错

```
示例：应使用 `joinRoom` 而不是 joinRoom。
```

3. 标题层级：标题必须按顺序使用（H1 → H2 → H3），不得跨级。一个标题下至少需要有两个子章节，不得出现孤儿章节。
```错误案例
# 一级标题
### 三级标题（错误：跳过了 H2）

# 一级标题
## 二级标题 A
### 三级标题 A1（错误：只有一个子章节）
```

4. 数及数量：表示数值范围时，使用半角短横线连接。数字带有国际单位和表示量的数词时，两个数字都要加上数词和单位。
```正确案例
10Mbps - 100Mbps
```

```错误案例
10 - 100Mbps
```

5. 链接检查：检查文档中的超链接是否正确。
- **检查项**：
  - 链接文案是否与目标内容一致（如链接文字是"LiveStore"但实际指向的 url 是 HostStore）
  - 外部链接是否有效、可访问（返回 404、403 等错误）
- **跳过检查**：
  - 内部链接（如 write.woa.com、内网域名）的**任何检查**，包括：
    - 可访问性检查（因为可能仅在特定网络环境下可访问）
    - 链接格式优化建议（如 URL 编码的锚点格式）
    - 链接风格建议


# 输出格式
请输出 JSON 格式，如果没有问题则输出空数组 `[]`。

**⚠️ 重要说明**：此 JSON 输出用于在内存中汇总审查结果，**不单独保存为文件**。最终会生成汇总的 JSON 文件（包含所有审查模块结果）和 Markdown 报告。**多个错误必须整合为同一个报错**

{
  "format_issues": [
    {
      "type": "空格/代码格式/标题层级/数值范围/链接错误",
      "severity": "P0/P1/P2",
      "category": "format",
      "location": "所在行或章节",
      "original": "原文内容",
      "description": "问题描述",
      "suggestion": "修改后的建议"
    }
  ]
}

注意事项：
1. **severity 判断标准**：
   - **P0（严重）**：链接失效（外部链接返回 404、403 等错误）、链接文案与目标内容完全不符
   - **P1（警告）**：标题层级、数值范围等格式问题（影响可读性但不影响功能）
   - **P2（建议）**：空格、代码格式问题（中英文空格、产品名与加粗标记间空格、行内代码与中文间空格等），属风格优化，不影响功能
2. **不检查**：数字与中文之间是否有空格（如 "5.0及以上"、"步骤1" 不报告）。加粗英文后直接跟中文不需要加空格（如 `**LiveKit**提供` 不报告）。
3. 对于空格、代码格式问题，如果文档中存在超过 3 处及以上问题，则仅输出 3 处错误，"suggestion"中增加"全文中有多处，请批量修改"。
4 不允许输出超过三个的同样问题
5 对于多个同类问题必须归结为一个报错，并在里面指出多个报错的具体位置；不允许输出多个错误
