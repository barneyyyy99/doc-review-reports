# 审查目标
审查技术文档中的描述是否自洽、完整、符合开发者认知流程。

# 审查原则

**核心原则**：按照下列审查规则来审核技术文档中的逻辑问题。

> **优先级标准**：本文档中的 P0/P1/P2 标注遵循 `skills/_shared/severity-standards.md` 定义的统一标准。

**⚠️ 零误报原则（最高优先级）**：
- **误报会让用户失去信任**，这比漏报更严重！
- 每个问题必须有确凿证据，不能基于推测、习惯、直觉判断
- 不确定的问题必须标记为 `【需人工确认】`，不能作为确定性问题输出
- **核心判断标准**：如果开发者能够通过上下文、常识或标准做法理解并完成任务，则不应标记为问题

# 审查规则

## 1. 前后描述矛盾
检查上下文描述是否自相矛盾：
- 前文要求参数为必填，示例代码中却没有该参数
- 前文介绍支持某功能，后文完全未说明如何实现
- 文档中对同一概念的描述存在冲突

## 2. 信息缺失
检查承诺的内容是否兑现：
- **严重**：前文明确承诺会介绍某功能/步骤，或标题承诺的核心内容，但后文完全没有相关内容（影响用户完成任务）
- **警告**：引用缺失，如提到"详见 XXX"但未提供链接（用户需要额外查找）
- **警告**：功能描述不完整，如只说"支持 A、B、C 功能"但未展开说明（不影响当前文档主线）

### 2.1 引用缺失（需明确边界）

**必须标记**：
- 提到具体文档名称（如"参考 主播开播 文档"、"详见 快速集成指南"）但未提供链接
- 使用"详见"、"参考"、"参见"等明确指向性词汇，但未提供链接
- 提到具体 API 文档、SDK 文档名称但未提供链接

**不标记**：
- 提到通用概念、常识性内容（如"参考 Android 官方文档"、"查看系统日志"）
- 文档中已提供链接，只是链接格式为内部文档系统格式（如 `write.woa.com` 域名的锚点链接）
- 文档名称出现在"下一步"、"相关文档"等章节，属于推荐阅读而非必需依赖

**修改建议格式**：
- ✅ **正确**：补充文档链接，如"请先参考 [主播开播](链接) 和 [观众观看](链接) 完成主流程的搭建"
- ❌ **错误**：在文档名称上加反引号（如"请先参考 `主播开播`"），这不是格式问题，而是链接缺失问题

**不检查的情况**：
- **文档内链接格式**：如果文档中使用了内部文档链接（如 `write.woa.com` 域名）指向文档内章节，即使链接格式为锚点 ID，也不标记为问题。这是文档系统的正常使用方式，链接的可用性由文档平台保证。

## 3. 逻辑断链（因果关系缺失）
要求用户执行操作但未说明原因，导致开发者不理解"为什么要这么做"。

**必须说明原因**：
- 操作违反直觉或常规做法（如删除默认配置、禁用系统功能）
- 操作有潜在风险或副作用（如修改系统配置、删除数据）
- 强制使用特定格式/值，但原因不明显（如"参数必须为 `GROUP${roomId}` 格式"）
- 关键配置项的取值会显著影响功能行为，但未说明选择依据

**无需说明原因**：
- 标准操作流程（安装依赖、导入包、创建实例、客户端数据筛选）
- 可选配置项（"功能进阶"等章节）
- 已有链接或前文已说明
- **示例代码中的布局参数、样式参数、UI 常量**（如坐标、宽高、颜色值等）
- **示例代码中显而易见的逻辑**（如 `prefix(9)` 取前9个、`% 3` 每行3个）
- **纯示例性质的硬编码值**，开发者显然需要根据自己的需求修改
- 不检查表格中的字段/文本结构

## 4. 步骤遗漏、跳跃或顺序错误
- 步骤 1 → 步骤 3，缺少关键步骤 2
- 后续步骤引用前文未说明的变量、对象或配置
- 依赖关系颠倒（如先调用 API，后初始化）

**必须标记**：
- 步骤编号跳跃（步骤 1 → 步骤 3），且缺少的步骤是关键操作
- 后续步骤使用了前文完全未提及的变量/对象/配置，且无法从上下文推断
- 操作顺序明显错误（如先调用方法，后创建实例）

**不标记**：
- 步骤编号跳跃，但文档已说明"跳过步骤 2"或"步骤 2 见其他文档"
- 后续步骤使用的变量/对象在前文已说明（即使不在同一章节）
- 示例代码中的占位符变量（开发者知道需要替换）
- 标准操作流程中的常见步骤（如"导入包"、"创建实例"），即使未明确列出

## 5. 关键分支场景缺失
仅关注**影响功能使用**的关键场景，不要求覆盖所有边缘情况。

**必须标记**：
- 提出"如果 XXX"条件，但只说明一种情况
- 多个互斥配置项，未说明选择依据
- 方法行为在不同场景下差异明显，但未说明

## 6. 条件表述冗余或不清晰
仅关注影响用户完成任务的关键情况。

**必须标记**：
- 使用"如果不...请忽略，如果..."（冗余，应直接用正向条件）
- 多层条件未覆盖常见组合，用户不知道自己属于哪种情况
- 并列方案无推荐项或适用场景，用户不知道选哪个

# 输出格式
请输出 JSON 格式，如果没有问题则输出空数组 `[]`。

```json
{
  "logic_issues": [
    {
      "type": "前后描述矛盾/信息缺失/逻辑断链/步骤遗漏/关键分支场景缺失/条件表述冗余或不清晰",
      "severity": "P0/P1",
      "category": "logic",
      "location": "问题所在章节",
      "original": "有问题的原文",
      "description": "问题描述",
      "suggestion": "修改建议"
    }
  ]
}
```

# 注意事项
1. **severity 判断标准**：
   - **P0（严重）**：前后矛盾、核心信息缺失（影响用户完成任务）、步骤遗漏/顺序错误、关键分支场景缺失（会导致开发者无法完成功能）
   - **P1（警告）**：逻辑断链（缺少因果说明）、引用缺失、条件表述冗余或不清晰（影响使用体验但不影响基本功能）
2. 对于同一类型的问题，如果文档中存在超过 3 处，则仅输出 3 处，在 "suggestion" 中增加"全文中有多处，请批量修改"。
3. **聚焦可操作性**：每个问题都必须能给出具体的修改建议，而不是泛泛而谈。
4. **避免重复**：共享层其他文件已检查的问题（如空格、代码语法）不要重复输出。
5. **避免过度审查**：
   - 示例代码是为了演示 API 用法，不是生产代码
   - 开发者具备基本的代码阅读能力和改造能力
   - 不要要求解释示例代码中的每个参数、每个常量
   - 只关注**真正会让开发者卡住或误解**的逻辑问题
6. **边界情况判断**：
   - 当不确定是否应该标记为问题时，优先选择**不标记**，或标记为 `【需人工确认】`
   - 如果开发者能够通过以下方式理解并完成任务，则不应标记：
     - 上下文推断
     - 标准库/框架的常识
     - 代码注释中的说明
     - 前文已提及的内容
7. **区分"引用缺失"和"信息缺失"**：
   - **引用缺失**：提到具体文档/资源名称但未提供链接（影响查找效率），修改建议应为补充链接，而非格式调整
   - **信息缺失**：技术概念/API/类名未说明来源或用途（影响理解和使用）
