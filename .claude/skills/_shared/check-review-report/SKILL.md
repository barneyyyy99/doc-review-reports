# 报告质量检查 Skill

## When to Use
- 在生成审查报告（Step 5）之后自动执行
- 作为质量保证环节，确保报告准确可靠

## Instructions

**⚠️ 核心原则**
- **零误报优先**：误报检查是最高优先级，必须严格执行
- **建设性反馈**：发现问题时提供具体的改进建议
- **不阻塞流程**：检查结果仅作为参考，不影响报告输出

---

## 检查流程

### Step 1：统计数据一致性检查（强制）

检查报告中的统计表格是否准确：

**必查项**：
1. P0 + P1 + P2 = 总计
2. 统计表中的数量 = 实际列出的问题数量
3. 批量问题标记准确性：只有 ≥3 个同类型问题才能标记"批量问题"

**检查方法**：
```
1. 从报告中提取统计表的数字
2. 实际数一遍每个优先级的问题数量
3. 对比是否一致
4. 如不一致，指出具体差异
```

---

### Step 2：误报检查（最高优先级）

验证报告中每个问题的准确性，确保符合"零误报原则"。

**必查项**：

1. **检查是否存在典型误报场景**：
   - ❌ 基于命名习惯推测 API 错误（未验证源码）
   - ❌ 未理解上下文就标记"代码片段不完整"
   - ❌ 将不确定的问题标记为 P0/P1（应标注"需人工确认"）
   - ❌ 把示例代码的简化当作错误（如占位符、省略的错误处理）

2. **验证 P0 问题的严重性**：
   - P0 问题必须真正影响功能使用
   - 不能将格式问题、风格问题标记为 P0
   - 确认问题有确凿证据（代码冲突、逻辑矛盾等）

3. **验证问题是否符合审查规则**：
   - 对照对应的审查规则文件（format-review.md、code-review.md 等）
   - 确认问题确实违反了规则中明确定义的检查项
   - 如果规则未覆盖，不应标记为问题，应标注为“需人工确认”）

4. **检查原文引用准确性**：
   - 原文内容是否准确无误
   - 是否有断章取义或理解偏差

**检查方法**：
```
对于报告中的每个问题：
1. 定位到原文档对应位置
2. 阅读完整上下文（前后至少 5 行）
3. 对照审查规则，确认是否真的违反规则
4. 如果发现误报，说明：
   - 为什么这不是问题
   - 可能被误判的原因
   - 如何避免类似误报
```

---

### Step 3：漏报检查

检查是否遗漏了应该发现的明显问题。

**重点检查项**（机械性检查项最容易漏报）：

1. **代码问题**：
   - 明显的语法错误（括号不匹配、分号缺失）
   - 同一 API 在文档中拼写不一致
   - 代码块中使用了未导入的类/函数

2. **逻辑问题**：
   - 前后描述明显矛盾
   - 步骤跳跃（步骤 1 → 步骤 3）
   - 承诺的内容完全未兑现

3. **错别字及中文病句。**

**检查方法**：
```
1. 快速扫描原文档，寻找明显问题
2. 对比审查报告，确认是否已发现
3. 如果发现漏报，说明：
   - 具体是什么问题
   - 应归类为什么类型（格式/代码/逻辑/语言）
   - 优先级应该是 P0/P1/P2
   - 为什么被遗漏（规则不够明确？检查逻辑有漏洞？）
```

**⚠️ 注意**：
- 不要过度要求，只关注**明显的、确定的**问题
- 不要标记模棱两可的问题
- 如果无法确定是否是问题，说明"建议人工复核"

---

### Step 4：报告质量检查

检查报告本身的质量是否符合标准。

**必查项**：

1. **问题描述清晰准确**：
   - 是否清楚说明了问题是什么
   - 是否说明了为什么这是问题
   - 是否提供了足够的上下文

2. **修改建议具体可操作**：
   - 是否提供了具体的修改方案
   - 修改建议是否与原文不同（避免"原文 → 原文"）
   - 是否说明了修改后的效果

3. **原文引用完整准确**：
   - 引用的代码片段是否完整
   - 引用的文字是否准确
   - 位置标注是否清晰（章节）

4. **批量问题处理正确**：
   - 只有 ≥3 个同类型问题才标记为批量
   - 批量问题是否列出了典型案例
   - 是否提供了批量修复建议

**检查方法**：
```
随机抽查 2-3 个问题，验证：
1. 描述是否让读者一眼就懂问题所在
2. 建议是否可以直接复制使用
3. 如有不足，指出具体改进方向
```

---

## Output Format（强制）

直接在对话中展示检查结果，使用以下格式：

```markdown
## 📋 报告质量检查结果

### ✅ 统计数据一致性
- 统计表：P0(2) + P1(3) + P2(0) = 总计(5) ✓
- 实际问题数：P0(2) + P1(3) + P2(0) = 5 ✓
- **结论**：统计数据准确

### 🔍 误报检查（最高优先级）
- 检查了 5 个问题，发现 0 个误报 ✓
- **结论**：所有问题均符合审查规则，无误报

### 🔎 漏报检查
- 发现 1 个潜在漏报：
  1. **[格式] 中英文空格缺失**
     - 位置：第 XX 行
     - 原文："LiveKit提供" 应为 "LiveKit 提供"
     - 分类：format_issues, P1
     - 原因：格式审查可能遗漏了此处

## 特殊情况处理

### 情况 1：发现严重误报

如果发现明显的误报（特别是 P0 误报），必须：
1. 立即指出具体问题
2. 说明为什么是误报
3. 建议从报告中移除该问题
4. 分析误报原因，建议规则改进

### 情况 2：发现严重漏报

如果发现明显的 P0 问题被遗漏，必须：
1. 详细说明问题内容
2. 解释为什么这是 P0 问题
3. 建议补充到报告中
4. 分析漏报原因，建议规则改进

### 情况 3：无法确定是否有问题

如果对某个判断不确定：
1. 诚实说明不确定
2. 建议"人工复核"

---

## 注意事项

1. **聚焦关键问题**：不要吹毛求疵，关注真正影响质量的问题
2. **提供建设性建议**：发现问题后要说明如何改进
3. **避免重复工作**：不要重新审查文档，只检查报告质量
4. **尊重审查规则**：只能根据现有规则判断，不能臆造新规则
