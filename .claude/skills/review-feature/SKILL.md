# 功能指南审查 Skill

## When to Use

- 功能使用指南、集成指南、最佳实践文档
- 包含完整使用流程的教程文档

## Instructions

### 共享层审查（强制执行）
以下审查规则来自 `skills/_shared/`，**必须在所有文档类型中执行**：

1. `logic-review.md`
2. `code-review.md`
3. `format-review.md`
4. `language-review.md`
5. `terms-review.md`

### 功能指南特有检查（L2 业务层）

#### 1. 时序流程完整性检查
**问题定义**：功能的生命周期是否完整？开发者能否从头到尾走通？

检查要点：
- **初始化阶段**：是否说明了前置条件（如必须先完成登录、必须先初始化 SDK）？如果引用了其他文档（如"前置条件详见快速集成文档"），则视为已说明。
- **配置阶段**：配置项是否有说明？必填/可选是否明确？
- **使用阶段**：核心 API 调用顺序是否正确？
- **销毁/清理阶段**：是否说明了资源释放（如取消监听、断开连接）？

**反例**：
```
❌ 文档只说"调用 startRecord() 开始录制"，但未说明：
   - 录制前是否需要先加入房间？（也没有引用说明前置条件的文档）
   - 录制结束后如何停止？
   - 停止后是否需要手动释放资源？

✅ 正确：
   "前置条件：需先完成快速集成（详见 [快速开始](link)）并加入房间。"
   → 明确引用了前置条件，不算流程不完整。
```

#### 2. 场景适配检查
**问题定义**：开发者如何知道什么场景用什么方案？

检查要点：
- 如果功能有多种实现方式（如本地录制 vs 云端录制），是否说明了适用场景？
- 如果配置项互斥（如 mode=auto 和 mode=manual），是否说明了选择依据？
- 是否提供了典型场景的完整示例（而不是散落的 API 说明）？

**反例**：
```
❌ "支持自动模式和手动模式"
   → 什么场景用自动？什么场景用手动？未说明。

✅ "自动模式适用于单人录制场景，SDK 自动选择最佳参数；
    手动模式适用于多人录制，需自行指定每个流的录制参数。"
```

#### 3. 依赖关系检查
**问题定义**：功能之间的依赖关系是否说明清楚？

检查要点：
- A 功能依赖 B 功能时，是否在 A 的文档开头说明了前置条件？
- 如果功能需要特定权限/配置，是否说明了如何获取？
- 如果功能依赖第三方服务（如云端 API），是否说明了如何配置？

**反例**：
```
❌ "调用 sendGift() 发送礼物"
   → 未说明发送礼物需要先加入房间、且房间类型必须支持礼物功能。
```

#### 4. 回调/事件处理检查
**问题定义**：异步流程是否说明清楚？开发者如何知道操作成功/失败？

检查要点：
- 异步操作（如 startRecord）是否说明了如何监听结果？
- 关键事件（如录制开始、录制失败）是否有回调说明？
- 回调参数是否有说明？

**反例**：
```
❌ "调用 startRecord() 开始录制"
   → 未说明如何知道录制是否成功启动。

✅ "调用 startRecord()，监听 onRecordStarted 事件确认录制已开始。
    如果启动失败，会触发 onRecordError 事件，错误码见下表。"
```

#### 5. 核心 API 参数缺少说明
**核心 API 定义**：文档实现章节中为了完成主要功能而必须调用的 API。

对于这些核心 API，正文中必须说明：
- 参数含义
- 参数类型
- 必填/可选
- 默认值（如有）
- 取值范围（如有）

不能仅通过代码注释介绍。非核心的辅助 API 可以只提供代码注释。

### 输出格式

请输出 JSON 格式，如果没有问题则输出空数组 `[]`。

**⚠️ 重要说明**：此 JSON 输出用于在内存中汇总审查结果，**不单独保存为文件**。最终会生成汇总的 JSON 文件（包含所有审查模块结果）和 Markdown 报告。

```json
{
  "feature_guide_issues": [
    {
      "type": "时序流程不完整/场景适配缺失/依赖关系缺失/回调处理缺失",
      "severity": "P0/P1",
      "location": "问题所在章节",
      "original_description": "有问题的原文",
      "description": "问题描述（说明为什么这是问题）",
      "suggestion": "修改建议（具体的补充内容）"
    }
  ]
}
```

### Severity 判断标准
- **P0（严重）**：缺失关键步骤、依赖关系未说明，会导致开发者无法完成功能
- **P1（警告）**：场景适配说明不清晰、回调说明不完整，影响使用体验但不影响基本功能

### 注意事项
1. **聚焦可操作性**：每个问题都必须能给出具体的修改建议，而不是泛泛而谈。
2. **避免重复**：共享层已检查的问题（如空格、代码语法）不要重复输出。
