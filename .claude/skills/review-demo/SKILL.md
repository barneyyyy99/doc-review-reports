# Demo 文档审查 Skill

## When to Use
- 审查"跑通 Demo"类型的文档
- 文档目标是让用户**开箱即用**、快速看到效果、验证开发环境


## Instructions

**⚠️ 零误报原则（最高优先级）**
- **误报会让用户失去信任**，这比漏报更严重！
- 每个问题必须有确凿证据，不能基于推测、习惯、直觉判断
- 不确定的问题必须标记为 `【需人工确认】`，不能作为确定性问题输出

---

### Step 0：执行共享层审查（强制）
在开始任何审核前，**必须完整执行以下共享审查规则**：

1. `logic-review.md`
2. `code-review.md`
3. `format-review.md`
4. `language-review.md`
5. `terms-review.md`

> 若共享层存在 FAIL 项，需记录，但仍继续执行后续步骤。

---

### Step 1：检查 Demo 介绍
**检查目标**：用户能否在开始操作前，了解 Demo 的基本信息和预期效果。

**核心原则：完整性检查，不评判质量**
- 只检查是否有 Demo 介绍和效果预览
- 不评判介绍的质量、详细程度、表述方式

**检查项：**
1. **产品 Demo 介绍**
   - 是否有 Demo 介绍文字（任何形式的介绍都可以）
   - ⚠️ **只检查有无，不检查质量**

2. **效果预览**
   - 是否提供 **Demo 运行效果的截图或图片**（让用户提前看到预期效果）
   - ⚠️ **本项为强建议项**：缺少会标记为 P1，但不是 P0

3. 如果有**在线 Demo**，检查链接是否有效

**判定为问题的情况：**
- 完全缺少产品介绍（用户不知道 Demo 是什么）→ **P1**
- 缺少效果预览图片（用户无法提前建立预期）→ **P1**
- 在线 Demo 链接失效（如已验证）→ **P0**

**不判定为问题的情况：**
- 介绍较简洁或较详细（只要有介绍即可）
- 介绍的表述方式、详细程度、亮点说明等质量问题
---

### Step 2：检查前提条件完整性

**核心原则：准备工作收口清晰**
- 本节是**准备阶段**，用户还未开始操作
- 重点检查**开通服务**和**环境准备**两大块

**检查项：**

#### 2.1 开通服务
1. **必要信息说明**
   - 是否明确列出需要获取的信息（如 **SDKAppID、SecretKey、UserSig** 等）
   - 是否说明如何获取（可以是直接在文档中介绍步骤，或者给开通服务文档的链接）

**判定为问题的情况：**
- 信息用途不清（用户不知道为什么要填这个）→ **P2**

**不判定为问题的情况：**
- 开通服务文档在其他位置（只要有链接即可）
- 信息已在后续"配置 Demo"章节说明（不重复要求）

#### 2.2 环境准备
1. **技术环境要求**
   - 是否明确**开发工具版本**（如 Xcode 14.0+、Android Studio 4.2+、Flutter 3.0+）
   - 是否明确**运行环境**（如 iOS 12.0+、Android 5.0+、Node.js 16+）
   - 是否明确**依赖项**（如 CocoaPods 1.11+、Gradle 7.0+）
   - 版本号是否具体（避免"较新版本"等模糊描述）
   - 是否说明操作系统要求（如 macOS 12+、Windows 10+）

**判定为问题的情况：**
- 环境要求缺失或模糊（如"需要安装 Flutter"但未说明版本）→ **P0**
- 特殊要求未说明（如必须真机但未提示）→ **P1**

**不判定为问题的情况：**
- 通用工具的安装方法（如"安装 Xcode"可假设用户知道）

---

### Step 3：操作步骤 - 获取 Demo
**检查目标**：用户是否能清楚地**获取 Demo 项目并找到正确的目录**。

**核心原则：位置明确、方式清晰**
- 必须说明**如何获取 Demo**（下载 or 克隆）
- 必须说明 **Demo 的具体目录位置**

**检查项：**
1. **获取方式**
   - 是否说明获取 Demo 的方式（如 Git 克隆、ZIP 下载、SDK 包内附带）
   - 是否提供仓库地址或下载链接

2. **目录位置说明**
   - ⚠️ **关键要求**：必须说明 **Demo 在仓库中的具体目录位置**（如 `TUIKit/Flutter/Demo/`）
   - 是否说明 Demo 的入口文件或目录结构（如"主程序位于 lib/main.dart"）

**判定为问题的情况：**
- 未说明如何获取 Demo（用户不知道去哪里下载）→ **P0**
- 未说明 Demo 的具体目录位置（用户下载后不知道打开哪个文件夹）→ **P0**
- 仓库/下载链接失效（如可验证）→ **P0**
---

### Step 4：操作步骤 - 配置 Demo
**检查目标**：用户是否能清楚地**找到配置文件并填写正确的配置信息**。

**核心原则：位置明确、格式清晰、安全提示**
- 必须说明**在哪个文件中填写配置**
- 必须说明**配置项的格式要求**

**检查项：**
1. **配置文件位置**
   - ⚠️ **关键要求**：必须明确说明**在哪个文件中填写配置**（如"打开 `lib/config.dart`"、"修改 `config/index.js` 文件"）
   - 是否提供文件路径（相对于 Demo 根目录）
   - 如有多个配置文件，是否逐一说明

2. **配置项说明**
   - 是否列出**必填配置项**（如 SDKAppID、SecretKey）
   - 是否说明配置项的**获取方式**（如"从控制台获取"、"在前提条件中已获取"）
   - 是否说明配置项的**格式要求**（如"纯数字"、"需要带引号"、"字符串类型"）

3. **多平台配置**
   - 如 Demo 支持多个平台（如 Flutter 同时支持 iOS 和 Android），是否分别说明各平台的配置方式

4. **安全提示**
   - ⚠️ **关键要求**：必须明确说明**仅用于 Demo 测试，不可用于生产环境**（如"UserSig 生成方式仅用于本地测试"）
   - 是否提示敏感信息的保护（如"不要将 SecretKey 提交到代码仓库"）

**判定为问题的情况：**
- 未说明配置文件位置（用户不知道在哪里填配置）→ **P0**
- 未说明必填配置项（用户不知道需要填什么）→ **P0**
- 缺少安全提示（用户可能在生产环境中误用测试配置）→ **P1**

**不判定为问题的情况：**
- 配置项获取方式已在"前提条件"中说明（不重复要求）
- 配置文件中已有清晰注释（文档可简化）

---

### Step 5：操作步骤 - 编译并运行 Demo
**检查目标**：用户是否能**按照步骤编译运行 Demo，并知道如何使用**。

**核心原则：步骤清晰、结果可验证、使用方法明确**
- 必须说明**如何编译和运行**（命令或界面操作）
- 必须说明**运行成功的标志**
- 必须说明**如何使用 Demo 的关键功能**

**检查项：**
1. **编译和运行步骤**
   - 是否说明**具体的编译命令或操作步骤**（如"执行 `flutter run`"、"点击 Xcode 的 Run 按钮"）
   - 是否说明在**哪个目录执行命令**（如"在 Demo 根目录执行"）
   - 是否说明运行环境（如"在真机上运行"、"在模拟器上运行"）

2. **运行成功标志**
   - 是否说明**运行成功的标志**（如"看到登录界面"、"控制台输出 'Server running'"）
   - 是否提供**运行成功的截图**（让用户对比确认）
   - 是否说明**关键功能的验证方式**（如"点击登录按钮后应看到主界面"）

3. **Demo 使用方法**
   - ⚠️ **关键要求**：必须说明**如何使用 Demo 的具体步骤**（如"如何登录账号"、"如何发送消息"、"如何加入房间"）
   - 是否说明核心功能的操作流程（不能只说"运行 Demo"就结束）
   - ⚠️ **禁止写"参考 Demo 的 README.md 文件"**（必须在本文档中说明）

4. **依赖安装**
   - 如需安装依赖（如 `npm install`、`pod install`、`flutter pub get`），是否说明
   - 是否说明依赖安装的**位置**（在哪个目录执行）

**判定为问题的情况：**
- 未说明编译或运行的具体命令/操作（用户不知道如何运行）→ **P0**
- 未说明运行成功的标志（用户无法判断是否成功）→ **P1**
- 未说明 Demo 使用方法（用户运行起来后不知道怎么操作）→ **P0**
- 需要安装依赖但未说明（用户可能运行失败）→ **P0**
- 步骤顺序错误（如先运行再安装依赖）→ **P0**

**不判定为问题的情况：**
- Demo 使用方法简单直观（如只需点击按钮，无需详细说明）
- 依赖安装已在"获取 Demo"步骤中说明（不重复要求）

---

### Step 6：常见问题
**检查目标**：用户遇到问题时，是否能**快速找到解决方案**。

**核心原则：覆盖典型场景、提供解决方案**
- 常见问题应基于**真实用户反馈**或**可预见的典型问题**
- 每个问题必须提供**现象 + 解决方案**

**检查项：**
1. **问题描述质量**
   - 每个问题是否有**清晰的现象描述**（如"报错：Module not found"）
   - 每个问题是否有**对应的解决方案**（不能只列问题不给答案）
   - 解决方案是否**可操作**（如"执行 `npm install` 重新安装依赖"）

3. **章节完整性**
   - 如文档未包含"常见问题"章节，标记为 **P2**（建议添加，但非强制）

**判定为问题的情况：**
- "常见问题"章节为空或只有标题（既然有这个章节，就应该填充内容）→ **P1**
- 问题只有现象没有解决方案（用户无法自行解决）→ **P1**
- 解决方案不可操作（如"检查网络"而未说明如何检查）→ **P2**

**不判定为问题的情况：**
- Demo 非常简单，确实没有常见问题（文档可不包含此章节）
- 常见问题已在其他文档中统一说明（如故障排查文档）

---


## Output Format（强制）

请输出 JSON 格式，如果没有问题则输出空数组 `[]`。

**⚠️ 重要说明**：此 JSON 输出用于在内存中汇总审查结果，**不单独保存为文件**。最终会生成汇总的 JSON 文件（包含所有审查模块结果）和 Markdown 报告。

```json
{
  "demo_issues": [
    {
      "type": "文档开头缺失/前提条件缺失/操作步骤不明确/常见问题缺失/联系方式缺失",
      "severity": "P0/P1/P2",
      "location": "问题所在章节（如"## 前提条件"、"### 配置 Demo"）",
      "original_description": "有问题的原文（如适用）",
      "description": "问题描述（为什么这是一个问题，会导致用户无法跑通 Demo 或体验受损）",
      "suggestion": "修改建议（具体的补充内容或修改方式）"
    }
  ]
}
```

**Severity 判定标准：**
- **P0**：会导致用户**无法跑通 Demo**（如缺少配置文件位置、缺少运行命令、链接失效）
- **P1**：会导致用户**操作困难或需要额外摸索**（如缺少效果预览、配置项格式不明确、缺少安全提示）
- **P2**：不影响跑通，但**影响体验或理解**（如缺少常见问题章节、话术不规范）
