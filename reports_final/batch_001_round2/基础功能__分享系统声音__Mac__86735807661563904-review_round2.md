## 6. 调整的问题（如有）

### P0-2 → 调整为 P1：回调描述与 API 命名矛盾（表述模糊）

**调整前**：P0 · 前后描述矛盾 — 「onSystemAudioLoopbackError 回调成功或失败的结果」

**调整后**：P1 · 表述模糊 — 「onSystemAudioLoopbackError 回调成功或失败的结果」命名与描述不一致，造成歧义

**调整原因**：

第一轮报告将此问题定级为 P0，依据是「API 命名与正文描述矛盾，开发者无法判断成功时是否触发此回调」。

经过复核，本问题性质为**表述模糊/命名歧义**而非**逻辑矛盾**：

1. 文档正文明确陈述「接口执行完成会通过 `onSystemAudioLoopbackError` 回调成功或失败的结果」— 这是文档的正面陈述，描述了 API 的实际行为（通过 errCode 值区分成功/失败，例如 errCode=0 表示成功，这在 TRTC 系列 SDK 中是常见模式）。
2. 从同类文档（Electron 版本）可佐证：`onSystemAudioLoopbackError` 的确以 `errCode===0` 表示成功，以其他错误码表示失败，与 Mac 文档描述的行为一致。
3. 该问题不会导致开发者「无法完成任务」（P0 核心判断标准），而是使开发者「理解不清晰但能通过进一步查阅 API 文档完成」（P1 判断标准）。

根据 severity-standards.md：
- P0：「导致理解错误导致实现完全错误」
- P1：「表述模糊，用户不确定如何操作」

此处正文表述虽因命名含 `Error` 而显得混淆，但未导致实现错误（文档自身说明了它同时回调成功和失败结果）。依据零误报原则，将此条由 P0 降级为 P1。

**调整后描述**：正文描述「onSystemAudioLoopbackError 回调成功或失败的结果」，但该回调名称含 `Error` 后缀，语义上仅指错误回调，开发者可能因命名而误以为成功时不会触发此回调。建议补充说明成功时的回调参数值（如 errCode=0 表示成功），或澄清该回调同时覆盖成功和失败。

**调整后建议**：补充说明：「成功时 errCode 返回 0，失败时 errCode 返回对应错误码（如 -1330、-1331、-1332 等）。」，消除命名带来的歧义。
