## 7. 新增的问题（如有）

以下问题在第一轮证据文件（`issues_deduped.json`）中均已记录，但未被写入最终报告，本轮补录：

### 补录 1 · [feature/logic] P0 · 观看屏幕分享章节缺少辅路接收方说明

**位置**：H2「观看屏幕分享」

**原文**：
> 若用户进行屏幕分享，且通过主流进行分享。房间里的其他用户会通过 TRTCCloudListener 中的 onUserVideoAvailable 事件获得这个通知。……希望观看屏幕分享的用户可以通过 startRemoteView 接口来启动渲染远端用户主流画面。

**问题描述**：
「观看屏幕分享」章节仅说明主路分享的接收方式（`onUserVideoAvailable` + `startRemoteView`），完全缺少辅路分享的观看方流程。而文档在 iOS 跨应用分享和 Windows 平台分享章节中均以辅路（`TRTCVideoStreamType.sub`）为推荐/主要方案，辅路观看方式的缺失会导致开发者按文档完成实现后远端无法看到屏幕分享内容。

**修改建议**：
在「观看屏幕分享」章节补充辅路分享接收端说明：若通过辅路分享，远端用户需监听 `TRTCCloudListener` 的 `onUserSubStreamAvailable` 事件，并调用对应的辅路渲染接口启动辅路画面渲染。

---

### 补录 2 · [logic] P1 · resumeScreenCapture 调用说明缺失

**位置**：H2「实现 Windows 平台屏幕分享」步骤3「开始屏幕分享」

**原文**：
> `pauseScreenCapture` 和 `stopScreenCapture` 的区别在于 pause 会停止屏幕内容的采集，并以暂停那一刻的画面垫片，所以在远端看到一直都是最后一帧画面，直到 resume。

**问题描述**：
文档说明了 `pauseScreenCapture` 的行为并提到"直到 resume"，但未给出 `resumeScreenCapture` 的调用说明，开发者知道可以暂停但不知道如何恢复，流程不完整。

**修改建议**：
在说明 `pauseScreenCapture` 后，补充"调用 `resumeScreenCapture` 可恢复屏幕采集"并提供 API 文档链接。

---

### 补录 3 · [feature] P1 · 前置条件未说明

**位置**：文档开头（无前置条件章节）

**原文**：
> （文档直接以「实现 Android 平台屏幕分享」H2 开篇，未说明前置条件）

**问题描述**：
文档未说明使用屏幕分享功能的前置依赖：开发者需已完成 TRTC SDK 的集成和初始化、已成功进入 TRTC 房间，才能调用屏幕分享相关 API。

**修改建议**：
在文档开头添加前置条件说明，如"使用屏幕分享前，请确保已完成 SDK 集成并已调用 `enterRoom` 成功进入 TRTC 房间。"

---

### 补录 4 · [feature] P1 · 各平台启动屏幕分享无回调确认说明

**位置**：全文（各平台启动屏幕分享章节）

**原文**：
> 要开启 Android 端的屏幕分享，只需调用 `TRTCCloud` 中的 `startScreenCapture()` 接口即可。（iOS/Windows 类似表述）

**问题描述**：
各平台启动屏幕分享章节均未说明如何监听屏幕分享启动成功/失败的回调（如 `onScreenCaptureStarted`、`onScreenCaptureStopped` 等），开发者无法得知分享是否成功启动。

**修改建议**：
在每个平台的启动屏幕分享章节后，补充回调监听说明，例如"可通过监听 `TRTCCloudListener` 的 `onScreenCaptureStarted`（分享已开始）、`onScreenCaptureStopped`（分享已停止）等回调确认分享状态。"

### 补录 5 · [terms] P2 · 辅路/主路术语描述与代码不一致

**位置**：H2「实现 Windows 平台屏幕分享」辅路/主路描述段
